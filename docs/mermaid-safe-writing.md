# Mermaid 安全写法对照表

这份文档专门记录在 VS Code Markdown preview 中编写 Mermaid 时的兼容性经验，避免把这类写图约束混到架构正文里。

目标只有三个：

1. 在 VS Code Markdown preview 里尽量不炸解析
2. 即使 Mermaid 版本不同，也尽量保持稳定
3. 让图的可维护性高于“语法花活”

## 高风险写法对照表

| 场景 | 高风险写法 | 更稳的写法 | 原因 |
| --- | --- | --- | --- |
| flowchart 节点里写字面量 `[]` | `A[text[]]` | `A["text[]"]` 或直接删掉 `[]` | `[]` 同时也是矩形节点定界符，容易把标签边界打乱 |
| flowchart 节点里写字面量 `{}` | `A[text{}]` 或 `B{check {x}}` | `A["text{}"]` 或把花括号改成普通文字 | `{}` 也是 Mermaid 常见形状/语法定界符，嵌在标签里风险高 |
| flowchart 节点换行 | `A[foo\nbar]` | `A[foo<br/>bar]` | `\n` 在不同 Mermaid 预览实现里不稳定，`<br/>` 更稳 |
| 虚线边上直接塞长句 | `A -. 说明文字很多 .-> B` | 尽量缩短文字，必要时再拆成说明节点 | dotted edge 的内联文本语法脆，长句最容易炸 |
| 节点标签里混复杂符号 | `A[a:b #c (d) [e]]` | `A["a:b #c (d) [e]"]` | 复杂字符最好整体加引号，别让解析器猜边界 |
| sequenceDiagram 深层分支 | 多层 `alt/else/end` 连续套娃 | 拆小图，或减少分支层级 | 不是语法必错，但维护时最容易把 `end` 配坏 |
| flowchart 里直接用保留词当 id | `end[结束]` | `EndNode[结束]` | Mermaid 对保留词和某些特殊 token 很敏感 |
| 边说明很多时 | 全塞进一条边标签里 | 优先缩短文案，必要时再拆图 | 图更稳，也更容易读 |

## 三条短规则

1. **节点文本复杂时，优先整体加引号。**
2. **flowchart 换行优先用 `<br/>`，不要用 `\n`。**
3. **说明性文字优先短写；只有在确实影响可读性时，才拆成独立说明节点。**

## 最短记忆版

> **Mermaid 里越像“代码字符”的东西，越不要裸写在节点标签边界上；复杂文本优先加引号，换行优先用 `<br/>`。**
